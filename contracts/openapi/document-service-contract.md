# Document Service — Contract

**Last Updated:** 2026-02-26  
**Language:** TypeScript · Node.js  
**Library:** PDFKit  
**Status:** Design / Pre-implementation

---

## Purpose

Accepts structured layout data and returns a formatted PDF document.
Owns all rendering logic — callers own their data and decide how to present it.

The service has no concept of domain types (transactions, game history, etc).
It understands only layout primitives: titles, headings, text blocks, tables, footers.
Any service can use it for any document with zero changes here.

```
[any service] → POST /document → [document-service] → PDF binary → [caller] → [client]
```

---

## Design Principles

- **Layout engine, not document factory**: callers shape their data into blocks, this service renders them
- **No domain knowledge**: transaction, game summary, player report — all look the same here: blocks
- **Stateless**: no storage, no queue. Request in, PDF out.
- **Binary response**: returns `application/pdf` directly. Caller streams to client.
- **TypeScript throughout**: consistent with auth-service.

## Endpoint

### POST /document

Generate a PDF document.

**Request:**
```json
{
  "caller": "string",
  "title": "string",
  "logo": "string (uuid)",
  "heading": "string",
  "sub_heading": "string",
  "blocks": [],
  "presentation": {}
}
```

**Field reference:**

| Field | Required | Notes |
|---|---|---|
| `caller` | Yes | Service name — must be in allowlist |
| `title` | Yes | Document title — appears in PDF header and filename |
| `logo` | No | UUID referencing an asset in game-db. **Not implemented** — field accepted, ignored. |
| `heading` | No | Large heading below title e.g. "Bank Transactions 2026-02-01 to 2026-02-26" |
| `sub_heading` | No | Smaller heading below heading |
| `blocks` | Yes | Ordered array of content blocks — see Block Types below |
| `presentation` | No | Styling overrides. **Not implemented** — field accepted, ignored. See Presentation below. |

---

## Block Types

Blocks are rendered in order. Each block is an object with exactly one key indicating its type.

### Text block
```json
{ "text": "string" }
```
Renders as a paragraph. Multiple consecutive text blocks render with standard paragraph spacing.

### Table block
```json
{
  "table": {
    "name": "string",
    "headers": ["col_a", "col_b", "col_c"],
    "rows": [
      "val1,val2,val3",
      "val4,val5,val6"
    ]
  }
}
```
`name` renders as a small heading above the table. `rows` are CSV strings — commas delimit columns, count must match `headers`. Commas within values must be quoted per RFC 4180.

### Footer block
```json
{ "footer": "string" }
```
Rendered at the bottom of the last page. Only the last footer block in the array is used if multiple are present.

## Full Request Example

```json
{
  "caller": "bank-service",
  "title": "Transaction History",
  "heading": "Bank Transactions 2026-02-01T00:00:00Z to 2026-02-26T23:59:59Z",
  "sub_heading": "PlayerOne",
  "blocks": [
    { "text": "The following transactions were recorded for this account during the specified period." },
    {
      "table": {
        "name": "Transactions",
        "headers": ["ID", "Type", "Amount", "Before", "After", "Timestamp"],
        "rows": [
          "442acc80,bet,50.00,1000.00,950.00,2026-02-26T21:09:02Z",
          "ddb9a06d,payout_loss,0.00,950.00,950.00,2026-02-26T21:09:13Z"
        ]
      }
    },
    { "text": "Net result: -50.00 chips." },
    { "footer": "Generated by TCA Blackjack · document-service" }
  ]
}
```

## Response (success)

```
HTTP 200
Content-Type: application/pdf
Content-Disposition: attachment; filename="<title>-<timestamp>.pdf"

<binary PDF data>
```

## Response (error)

```
HTTP 4xx/5xx
Content-Type: application/json

{
  "error": {
    "code": "string",
    "message": "string"
  }
}
```

---

## Error Codes

| Code | Meaning |
|------|---------|
| `unknown_caller` | Caller not in service allowlist |
| `missing_required_fields` | `caller`, `title`, or `blocks` absent |
| `invalid_block` | Block has unknown type or malformed structure |
| `invalid_table_rows` | Row column count doesn't match headers |
| `render_failed` | PDFKit threw an error during generation |

## Presentation (Reserved)

The `presentation` block is accepted and validated by the service but not yet acted upon.
All documents currently render with default styling. Field is reserved for future
user-controlled formatting. Callers may begin sending it — the contract will not change
when presentation control is implemented.

Planned fields:
```json
{
  "presentation": {
    "font": "string",        // Font family e.g. "Helvetica", "Courier" — default: "Helvetica"
    "colors": {
      "primary": "string",   // Hex — headers, accents — default: "#1f6feb"
      "secondary": "string", // Hex — subheadings, borders — default: "#8b949e"
      "text": "string"       // Hex — body text — default: "#24292f"
    },
    "page_size": "string"    // "A4" | "Letter" — default: "Letter"
  }
}
```

---

## Logo (Reserved)

The `logo` field accepts a UUID referencing an asset record in game-db.
**Not implemented** — field is accepted, ignored at render time.

When implemented: document-service will call game-state to resolve the UUID to image bytes,
embed the image in the PDF header. No contract changes required.

## Service Allowlist

```
bank-service, game-state, auth-service
```

Unknown callers are rejected with `unknown_caller`.

## What's Stubbed (PoC)

Nothing. This service is simple enough to implement fully:
- Real PDF generation via PDFKit
- Real caller allowlist enforcement  
- Real block validation
- In-memory counter for health endpoint

Extension points:
- New block types: add a renderer function, register it — zero pipeline changes
- Presentation control: PDFKit already accepts fonts and colors, just wire `presentation` fields through
- Logo: one lookup call to game-state, one PDFKit image embed
